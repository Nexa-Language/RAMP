RAMP Paper Draft Complete: A New Paradigm for Serial Agent Evaluation

We're excited to announce that the first complete draft of the RAMP paper is ready. The paper introduces a fundamentally new approach to evaluating AI coding agents — one that measures not just what agents can solve, but how they evolve through a dependency chain.

📄 Paper Structure (6 Sections)

  1. Introduction — The problem with static benchmarks
  2. Related Work — Comparison with 8 existing benchmarks
  3. Benchmark Design — Task chain, resurrection protocol, metrics
  4. Experiment Setup — 22 models, 4 agent backends
  5. Results & Analysis — Leaderboard, resurrection gains, process diagnostics
  6. Discussion & Conclusion — Limitations, future directions

The Core Insight: Static Benchmarks Miss Cascading Failure

Existing benchmarks like SWE-bench, HumanEval, and GAIA evaluate agents on independent, isolated tasks. Each task is a clean sandbox — the agent starts fresh, solves one problem, and moves on. This design enables reproducibility but overlooks a fundamental property of real engineering:

In practice, software development proceeds along dependency chains where later stages build upon the artifacts of earlier ones. When an agent fails at an early stage, the entire downstream pipeline becomes unreachable.

Our data proves this is not a theoretical concern. Only 2 out of 22 model configurations can complete the full 6-task compiler pipeline without external correction. The median zero-shot pipeline depth is just 1 task.

Three Key Innovations

1. Serial Task Structure with Artifact Dependencies

RAMP is grounded in compiler construction — a domain with strict causal dependencies. Agents must progress through six tasks:

TaskNameInput Artifact
T0Environment Setup—
T1Lexer (Tokenization)Preprocessed source
T2Parser (AST Construction)Token stream (from T1)
T3IR Generation (LLVM IR)AST/JSON (from T2)
T4IR Optimization (LLVM Pass)Raw IR (from T3)
T5Code Generation (RV64 Assembly)Optimized IR (from T4)

Each task's output becomes the next task's input. If T1 fails, T2-T5 receive broken artifacts and typically cascade to zero scores — regardless of the agent's downstream capability.

2. Resurrection Mechanism

This is the paper's most important methodological contribution. When an agent fails a task, RAMP's Resurrection mechanism injects the golden intermediate artifact (the correct output from the reference implementation), allowing the agent to continue to downstream tasks.

Resurrection decomposes the compound failure "cannot reach Task k+1" into two independent measurements:

  • Predecessor failure: the agent could not produce a passing artifact at Task k
  • Downstream capability: given correct prerequisites, can the agent succeed at Task k+1?
📊 The Numbers

Resurrection is not a theoretical nicety — it's empirically essential. 17 out of 19 configurations gain >40 pipeline score points with resurrection. The median gain is 73.65 points. Without resurrection, most agents can only be evaluated at 1-2 tasks before cascading failure blocks everything downstream.

3. Process-Centric Evaluation

Beyond final scores, RAMP systematically records tokens, interaction turns, shell/build/test retries, elapsed time, and failure types. These process metrics are first-class analytical objects alongside outcome scores.

A key finding: process cost does not predict outcome (R² = 0.007). Agents that invest more computational budget do not systematically achieve better results. This decoupling is itself diagnostic — it reveals that process metrics capture qualitatively distinct information from outcome metrics.

Key Empirical Findings

  • Serial evaluation preserves overall rankings (Kendall τ = 0.763, p < 0.001) but reveals specific failure patterns invisible to static benchmarks
  • Agent scaffold is an independent variable: the same model (mimo-v2.5-pro) scores 67.04 on OpenHands vs. 98.23 on Kimi CLI — a 31-point gap
  • Process cost is decoupled from outcome: R² = 0.007 for build count → score regression
  • Failure taxonomy reveals 5 distinct failure modes: predecessor failure, task-local reasoning failure, execution failure, process collapse, and cost overrun

Limitations and Future Work

We are transparent about the paper's current limitations:

  • Single-run evaluation: each configuration is evaluated once; multi-seed evaluation is planned
  • Domain specificity: the methodology is demonstrated on compiler construction; generalization to other serial workflows (ML pipelines, DevOps) requires further study
  • Contamination risk: YatCC is a public repository; future versions may use private codebases or temporal splitting
  • Task count: 6 tasks provide limited statistical power, though structural diversity partially compensates

What This Means for the Field

RAMP represents a paradigm shift in agent evaluation. The key insight is not that serial benchmarks are "harder" — it's that they measure a fundamentally different evaluation axis. Static benchmarks answer "can the agent solve task k?" Serial benchmarks answer "can the agent progress along the dependency chain, and if not, where and why?"

This distinction matters because real engineering is serial. When you deploy an AI coding agent in production, it doesn't solve isolated LeetCode problems — it builds systems where each component depends on the last. RAMP is the first benchmark designed to evaluate agents in this more realistic setting.

🔮 Looking Ahead

We're actively working on multi-seed evaluation, cross-benchmark correlation with SWE-bench, and evolutionary gain measurement (comparing agent performance with self-produced vs. golden artifacts). The paper will be submitted to a top-tier venue in the coming weeks.
