EvoBench First Results: 19 Models Evaluated on Compiler Construction

Executive Summary

We ran the first large-scale evaluation of 19 LLMs on EvoBench using OpenHands SDK (a real agent framework with terminal, file editor, and task tracker). The results reveal a clear capability cliff between "understanding" and "implementing" compilers.

💡 Key Finding

deepseek-v4-pro is the only model that achieved 100% on Tasks 0-3 without any resurrection on Task 3. This means it genuinely understood the compiler pipeline, not just copied patterns from golden answers.

The Task 4/5 Wall

Every single model scored 0% on Task 4 (IR Optimization) and Task 5 (RV64 Backend). This is not a bug — these tasks require:

  • Task 4: Implementing custom LLVM optimization passes (ConstantFolding, Mem2Reg, etc.) and achieving better performance than clang -O2
  • Task 5: Implementing 4 specific functions in EmitMIR.cpp for RV64 instruction selection

The gap between Task 3 (100% achievable) and Task 4 (0% for all) represents a massive capability cliff. This is where EvoBench truly differentiates itself from simpler benchmarks.

Resurrection Mechanism Analysis

The resurrection mechanism worked as designed:

  • Models that failed Task 1 (Lexer) were resurrected and could attempt Task 2 (Parser)
  • Several models achieved 100% on Task 2 after resurrection, proving they can work with golden inputs
  • The Evolutionary Gain metric (not yet computed) will quantify how much better agents do with their own code vs. golden code

Metrics Formula

We introduced two key metrics:

  • Mean Reward: Weighted average score with 1.2x bonus for direct (non-resurrected) passes
  • Pass Score: Weighted pass rate with 1.5x bonus for direct passes
Mean Reward = Σ(score[i] × weight[i] × bonus[i]) / Σ(weight[i])
weight = [1, 2, 2, 3, 3, 4]  // Task 0-5, increasing importance
bonus = 1.2 (no resurrection) / 1.0 (resurrected)

What's Next

  • Add more agent backends (Claude Code, Gemini CLI, Codex)
  • Compute Evolutionary Gain with paired experiments
  • Expand to more models as they become available

Why YatCC Is the Perfect Substrate for an Agent Benchmark

The Problem with Existing Benchmarks

Current agent benchmarks like SWE-bench, GAIA, and AgentBench share a fundamental limitation: they test isolated, atomic tasks. Each task starts from a clean slate. There's no concept of "building on previous work" or "learning from mistakes."

This is like testing a programmer by giving them 100 independent LeetCode problems. It measures instantaneous coding ability, not engineering evolution.

Why Compiler Construction?

Compiler construction is uniquely suited for evaluating agent evolution because:

1. Strong Causal Dependencies

You cannot write a parser without a working lexer. You cannot optimize IR without generating it first. This creates a natural serial pipeline where each task builds on the previous one.

2. Objective, Automated Grading

Unlike code review or documentation tasks, compiler correctness is binary: does the compiled program produce the correct output? Yes or no. This eliminates subjective evaluation.

3. Graduated Difficulty

The 6 tasks span from trivial (Task 0: environment setup) to extremely challenging (Task 5: RV64 backend). This creates a natural difficulty curve that can differentiate between models.

4. Real-World Relevance

Every programmer uses compilers. This isn't an artificial benchmark — it's a real engineering task that CS students spend a full semester on.

YatCC's Unique Advantages

YatCC (中山大学编译原理课程实验) has several features that make it ideal:

  • Complete test infrastructure: 73 test cases with automated scoring scripts
  • Resurrection mechanism: Built-in support for injecting golden answers when tasks fail
  • CMake-based build: Standard build system that any agent can understand
  • Graduated scaffolding: Each task provides starter code that agents can extend
  • Open source: Available on GitHub for reproducibility

The Resurrection Innovation

The key innovation of EvoBench is the Resurrection Mechanism. When an agent fails at Task N:

  1. The benchmark controller modifies config.cmake to set TASK{N}_REVIVE=ON
  2. CMake reconfigures, injecting golden answers as input for Task N
  3. The agent can continue to Task N+1 with correct inputs

This solves the cascading failure problem: without resurrection, a single failure at Task 1 would make it impossible to evaluate Tasks 2-5.

"The resurrection mechanism transforms EvoBench from a pass/fail gate into a diagnostic tool. We can see exactly where each model's capabilities break down."
April 20, 2026 Vision

EvoBench: A Vision for Testing Agent Evolution

The Core Question

Can AI agents evolve? Not in the biological sense, but in the engineering sense: can an agent learn from its previous work, accumulate knowledge, and apply it to increasingly complex tasks?

Current benchmarks can't answer this question. They test agents in isolation — each task is a fresh start. But real software engineering is cumulative. A senior engineer isn't someone who can solve 100 independent problems; it's someone who can build a complex system over months, learning and adapting along the way.

The EvoBench Concept

EvoBench is designed to test serial agent evolution:

  • Sequential tasks: Task 1 → 2 → 3 → 4 → 5, each building on the previous
  • Resurrection on failure: When an agent fails, inject golden answers and continue
  • Multi-dimensional metrics: Not just pass/fail, but how well the agent evolves

Inspiration

The idea came from observing that:

  1. Existing benchmarks (SWE-bench, GAIA) test isolated tasks
  2. Voyager (Nvidia) showed that agents can build skill libraries in Minecraft
  3. Compiler construction has natural serial dependencies and objective grading
  4. 中山大学's YatCC course has a complete test infrastructure with resurrection support

What We Want to Measure

  • Zero-shot Pipeline Pass Rate: Can the agent complete all tasks without resurrection?
  • Node Pass Rate: Given correct inputs, can the agent complete each individual task?
  • Average Resurrection Count: How many times does the agent need golden answers?
  • Evolutionary Gain: Does the agent perform better with its own code vs. golden code?

Next Steps

Our immediate plan:

  1. Build the evaluation infrastructure (Docker sandbox, CMake automation, score parsing)
  2. Implement the resurrection mechanism
  3. Create agent tool interfaces (file read/write, command execution)
  4. Run the first evaluation with multiple models
  5. Publish results and invite the community to participate
"We want to test agents the way we test students: through a semester-long project that requires building on previous knowledge."
